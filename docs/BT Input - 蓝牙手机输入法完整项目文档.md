# 文档 1：产品定义说明书

## 1.1 产品概述

**BT Input** 是一款将手机变为 Windows 电脑无线输入设备的工具。用户通过蓝牙将手机与 PC 配对后，在手机上使用**任意系统输入法**（拼音、语音、手写、五笔等）输入文字，文字**实时同步**到 PC 当前光标所在位置。

PC 端表现为一个仿 IME（输入法）浮动状态条程序，手机端为一个跨平台 App（Android + iOS），两者通过 BLE（低功耗蓝牙）直连，无需 WiFi 或互联网。

## 1.2 要解决的问题

| 痛点 | 描述 | 影响人群 |
|------|------|---------|
| PC 中文输入效率低 | 很多用户在 PC 上打字速度远不如手机 | 全体中文 PC 用户 |
| 语音输入缺失 | Windows 原生语音输入体验差，第三方方案需联网 | 需要大量文字录入的用户 |
| 多语言切换繁琐 | PC 上切换中/英/日/韩输入法操作繁琐 | 多语言工作者 |
| 手写输入不便 | PC 端没有触摸屏时无法手写输入 | 手写偏好用户、老年用户 |

## 1.3 目标用户画像

### 用户画像 A — "高效文字工作者" 小李

- 28 岁，内容创作者
- 每天需要在 PC 上输入大量中文文本
- 手机上使用讯飞语音输入法，一分钟可口述 200+ 字
- 痛点：PC 上没有好用的语音输入方案
- 期望：对着手机说话，文字直接出现在 PC 的 Word 文档里

### 用户画像 B — "多语言工程师" 王工

- 35 岁，软件开发者
- 工作中频繁切换中英文编写文档和代码注释
- 手机输入法支持中英混输且准确率高
- 痛点：PC 上中英切换干扰编码节奏
- 期望：用手机输入中文注释，英文仍用 PC 键盘

### 用户画像 C — "长辈用户" 张阿姨

- 58 岁，不熟悉拼音
- 手机上使用手写输入法
- 偶尔需要在 PC 上填写表格、发邮件
- 痛点：PC 上完全不会打字
- 期望：手机手写，PC 出字

## 1.4 产品定位与差异化

```
             独立于网络          专注文字输入          利用手机已有输入法
                │                    │                      │
    ┌───────────┴───────────┐       │          ┌───────────┴───────────┐
    │                       │       │          │                       │
    ▼                       ▼       ▼          ▼                       ▼
  BT Input               竞品对比
  蓝牙直连               Microsoft Phone Link → 需要 WiFi，无文字输入功能
  无需网络               KDE Connect           → 仅限 Android + Linux
  零延迟                 Unified Remote        → 侧重遥控，非文字输入
                         各类远程桌面           → 需要网络，延迟高
```

**核心差异化**：

1. **蓝牙直连** — 不依赖 WiFi 或互联网
2. **借力系统 IME** — 不重新造输入法轮子，用户手机上的搜狗/讯飞/Gboard 都能直接用
3. **实时同步** — 打一个字出一个字，体验等同本地输入法
4. **仿 IME 体验** — PC 端表现得像一个系统输入法，而非一个独立 App

## 1.5 产品价值主张

> **一句话**：让你手机上最顺手的输入法，成为你电脑上最好用的输入法。

## 1.6 商业模式（远期）

| 阶段 | 模式 | 说明 |
|------|------|------|
| Phase 1 | 免费开源 | 积累用户和口碑 |
| Phase 2 | Freemium | 基础功能免费，高级功能（多设备、云剪贴板同步）付费 |
| Phase 3 | 企业版 | 为企业提供多设备管理、安全审计功能 |

# 文档 2：需求定义文档 (PRD)

## 2.1 背景与目标

### 背景

参见 [产品定义说明书 §1.2 要解决的问题](#12-要解决的问题)。

### 目标

| 目标 | 衡量指标 |
|------|---------|
| 核心功能可用 | 用户可通过手机输入中文，实时出现在 PC 记事本中 |
| 输入延迟可接受 | 单字输入端到端延迟 < 100ms (P95) |
| 大段文本不卡顿 | 一次性输出 30 字以内无可感知延迟 |
| 长时间使用稳定 | 连续使用 2 小时不断连、不丢字 |
| 跨平台手机支持 | 同时支持 Android 12+ 和 iOS 15+ |

## 2.2 需求优先级定义

| 优先级 | 定义 |
|--------|------|
| **P0** | 必须实现 — 没有这个功能产品无法使用 |
| **P1** | 核心体验 — 缺失会严重影响用户满意度 |
| **P2** | 体验增强 — 有显著正向影响，应当实现 |
| **P3** | 锦上添花 — 未来迭代中考虑 |

## 2.3 功能需求

### 2.3.1 蓝牙连接管理

| 优先级 | 需求 | 验收标准 |
|--------|------|---------|
| P0 | 手机 App 能够扫描并连接到 PC 端程序 | 在蓝牙范围内（10 米），手机可发现 PC 端服务并完成配对，耗时 < 10 秒 |
| P0 | 连接断开后自动重连 | 蓝牙信号恢复后 5 秒内自动重连，无需用户操作 |
| P1 | 连接状态实时显示 | 手机和 PC 双端均可看到当前连接状态（已连接/连接中/已断开） |
| P1 | 首次配对引导 | 首次使用时，PC 端和手机端均有引导流程帮助用户完成配对 |
| P2 | 记住已配对设备 | 下次打开 App 后自动连接上次配对的设备 |
| P3 | 支持多设备管理 | 一台手机可保存多台 PC 的配对信息，手动切换 |

### 2.3.2 文本输入核心功能

| 优先级 | 需求 | 验收标准 |
|--------|------|---------|
| P0 | 手机输入框文本变化实时同步到 PC | 手机输入框每次文本变化，PC 端对应窗口光标处实时出现相同文字 |
| P0 | 支持中文输入 | 使用拼音、手写、语音等任意输入法输入的中文均能正确同步 |
| P0 | 支持英文和数字输入 | 英文字母、数字、常见符号均能正确同步 |
| P0 | 支持删除操作 | 手机端删除字符时，PC 端同步删除 |
| P1 | 支持一次性大段文本 | 语音输入一次性产出 30 字以上文本时，PC 端流畅显示，无丢字 |
| P1 | 支持换行/回车 | 手机端按回车键时，PC 端同步输入换行 |
| P2 | 支持特殊按键 | Tab、方向键等特殊按键可通过手机端发送 |
| P2 | 支持 emoji 输入 | 手机端输入的 emoji 能在 PC 端正确显示 |
| P3 | 支持剪贴板同步 | 手机端复制的文本可同步到 PC 剪贴板 |

### 2.3.3 PC 端体验

| 优先级 | 需求 | 验收标准 |
|--------|------|---------|
| P0 | PC 端以系统托盘图标常驻 | 程序启动后显示在系统托盘，不占用任务栏 |
| P0 | 全局快捷键激活/停用 | 用户可通过快捷键（默认 Ctrl+Shift+B）切换 BT Input 的启用/停用状态 |
| P0 | 文字注入到当前焦点窗口 | 手机输入的文字出现在当前前台窗口的光标位置 |
| P1 | 浮动状态条 | 激活状态下显示仿 IME 的浮动条，显示连接状态和临时文本预览 |
| P1 | 浮动条跟随光标 | 浮动条位置跟随当前输入光标移动 |
| P1 | 浮动条不抢焦点 | 浮动条窗口不会从其他应用抢走焦点 |
| P2 | 自定义快捷键 | 用户可在设置中修改全局快捷键 |
| P2 | 开机自启动 | 可设置为开机自动启动 |

### 2.3.4 手机端体验

| 优先级 | 需求 | 验收标准 |
|--------|------|---------|
| P0 | 输入框自动拉起系统输入法 | 打开 App 并连接后，输入框自动获得焦点并弹出系统输入法 |
| P0 | 支持 Android 和 iOS | Android 12+ 和 iOS 15+ 均可正常使用 |
| P1 | 输入框自动清空 | 输入内容超过 500 字符且用户 2 秒无操作时，自动清空输入框，PC 端不受影响 |
| P1 | 连接状态显示 | 清晰显示当前与 PC 的连接状态 |
| P2 | 深色模式支持 | 自动跟随系统深色/浅色模式 |
| P2 | 输入统计 | 显示本次会话已输入字符数 |

## 2.4 非功能需求 (NFR)

| 类别 | 需求 | 指标 |
|------|------|------|
| **性能** | 单字输入端到端延迟 | < 100ms (P95) |
| **性能** | 大段文本（30 字）注入时间 | < 500ms |
| **可靠性** | 连续使用不丢字 | 2 小时内丢字率 < 0.01% |
| **可靠性** | 自动重连成功率 | > 99% |
| **兼容性** | PC 操作系统 | Windows 10 1903+ / Windows 11 |
| **兼容性** | 手机操作系统 | Android 12+ / iOS 15+ |
| **兼容性** | PC 应用兼容性 | 记事本、Word、VS Code、浏览器输入框等主流应用 |
| **安全性** | 蓝牙通信 | 使用 BLE 加密配对，传输数据不含敏感信息 |
| **资源占用** | PC 端内存 | < 80 MB |
| **资源占用** | 手机端电量 | 连续使用 1 小时耗电 < 5% |
| **安装** | PC 端分发 | 单文件 exe，无需安装 .NET Runtime |
| **安装** | 手机端分发 | APK 直接安装 / App Store / Google Play |

## 2.5 版本路线图

### Phase 2 — 体验打磨（~3 周）

### Phase 3 — iOS + 产品化（~4 周）

- iOS 端适配
- 长时间连续输入测试（2 小时+）
- 安装包优化 / 应用商店上架
- 深色模式 / 用户引导

|------|------|---------|

# 文档 3：UI/UX 设计书

## 3.1 设计原则

1. **极简** — 手机端只有一个核心界面，PC 端只有一个托盘图标和浮动条
2. **无感** — 用户的注意力应始终在 PC 屏幕上，手机端是"放在桌上不用看"的外设
3. **仿 IME** — PC 端的视觉和交互行为应尽量接近系统输入法

### 3.3.2 浮动状态条 (Floating Bar)

浮动条是 PC 端最核心的 UI 元素，模拟输入法候选栏的视觉效果。

```
状态 1：待机状态（已连接但无输入）
┌─────────────────────────────────────┐
│  🔵 BT Input · 已连接               │   半透明深色背景
└─────────────────────────────────────┘   圆角 8px
                                          跟随光标位置

状态 2：正在输入（有文本同步中）
┌─────────────────────────────────────┐
│  🟢 正在输入...  今天天气             │   绿色脉冲指示
└─────────────────────────────────────┘   显示最近同步的文本预览

状态 3：连接断开
┌─────────────────────────────────────┐
│  🔴 连接已断开 · 等待重连...          │
└─────────────────────────────────────┘

状态 4：已停用（快捷键停用后）
  （浮动条隐藏，仅保留托盘图标）
```

**视觉规格**：

| 属性 | 值 |
|------|------|
| 窗口类型 | Tool Window（不在任务栏和 Alt+Tab 中显示） |
| 窗口样式 | `WS_EX_NOACTIVATE` + `WS_EX_TOPMOST` + `WS_EX_TOOLWINDOW` |

## 4.3 数据流详解

```
完整的数据流（一个字符从手机到 PC 的生命周期）:

  T=0ms    用户在手机输入框输入一个字
           │
  T=0ms    Flutter TextField.onChanged 触发
           │
  T=0ms    ThrottledDiffSender 收到新文本
           │  首次变化 → 立即处理
           │  后续变化 → 50ms 节流窗口
           │
  T=1ms    DiffEngine.computeDelta(oldText, newText)
           │  使用前缀+后缀算法计算增量
           │  产出: TextDelta { op: APPEND, text: "好" }
           │
  T=2ms    ProtocolEncoder.encode(delta)
           │  序列化为 JSON: {"t":1,"s":42,"o":"A","d":"好"}
           │
  T=3ms    BLE GATT Notify → 发送到 PC
           │  如果数据 > MTU → 分包发送
           │
  T=10ms   BLE 传输延迟（典型 7-15ms）
           │
  T=15ms   PC BLE Central 收到通知
           │
  T=16ms   ProtocolDecoder.decode(bytes)
           │  反序列化 JSON → TextDelta 对象
           │
  T=17ms   TextInjectionEngine.handleMessage(delta)
           │  判断注入策略:
           │  - 短文本(≤10字) → SendInput(KEYEVENTF_UNICODE)
           │  - 长文本(>10字) → Clipboard + Ctrl+V
           │
  T=18ms   SendInput 执行
           │  文字出现在 PC 前台窗口光标处
           │
  T=18ms   FloatingBar 更新预览文本
           │
  T≈20ms   用户看到 PC 上出现文字 ✅

  端到端延迟: ~20ms（远低于 100ms 目标）
```

## 4.7 错误处理与容错

| 场景 | 检测方式 | 处理策略 |

# 文档 5：BLE 通信协议文档

## 5.1 协议版本

| 字段 | 值 |
|------|------|
| 协议名称 | BT Input Protocol (BTIP) |
| 版本 | 1.0 |
| 传输层 | BLE GATT |
| 编码格式 | JSON (v1.0) / Binary (v2.0 预留) |
| 字符编码 | UTF-8 |

## 5.2 协议分层

```
┌──────────────────────────────────┐
│  Layer 3: Application            │  消息语义：TextDelta, Control, Status
├──────────────────────────────────┤
│  Layer 2: Session                │  序列号, 心跳, 同步机制
├──────────────────────────────────┤
│  Layer 1: Transport              │  BLE 分包/重组, MTU 适配
├──────────────────────────────────┤
│  Layer 0: BLE GATT               │  Service, Characteristics, Notify/Write
└──────────────────────────────────┘
```

## 5.3 GATT Service 定义

| 元素 | UUID | 方向 | 属性 | 说明 |
|------|------|------|------|------|
| **Service** | `0000FFF0-0000-1000-8000-00805F9B34FB` | — | — | BT Input Service |
| **Text Char** | `0000FFF1-...` | Phone → PC | NOTIFY | 文本数据通道 |
| **Control Char** | `0000FFF2-...` | PC → Phone | WRITE | 控制指令通道 |
| **Status Char** | `0000FFF3-...` | Phone → PC | NOTIFY | 状态/心跳通道 |

## 5.4 Transport 层 — 分包协议

当消息总长度超过 MTU 可用 payload 时，启用分包：

```
包头结构 (3 bytes):

  Byte 0: msg_id (uint8)     — 消息 ID，同一消息所有包相同
  Byte 1: high 4 bits = seq  — 当前包序号 (0-15)
          low  4 bits = flags
            bit 0: 1 = 最后一包
            bit 1: 保留
  Byte 2: total (uint8)      — 总包数

  Payload: 剩余 bytes (MTU - 3 bytes header)
```

**分包示例**（MTU=100，消息 250 bytes）：

| 包序号 | msg_id | seq | last | total | payload 长度 |
|-------|--------|-----|------|-------|-------------|
| 1 | 7 | 0 | 0 | 3 | 97 |
| 2 | 7 | 1 | 0 | 3 | 97 |
| 3 | 7 | 2 | 1 | 3 | 56 |

PC 端收齐所有包后按 seq 顺序拼接 payload，然后交给 Session 层处理。

## 5.5 Session 层 — 序列号与可靠性

### 序列号规则

- 手机端维护全局递增 `tx_seq: uint16`（0 → 65535 → 0 循环）
- 每发一条 Application 层消息，`tx_seq++`
- PC 端维护 `expected_seq`，检测是否连续

### 可靠性策略

```
正常流程:
  Phone ──[seq=1, TextDelta]──▶ PC    ✅
  Phone ──[seq=2, TextDelta]──▶ PC    ✅

丢包场景:
  Phone ──[seq=3, TextDelta]──✕       丢失
  Phone ──[seq=4, TextDelta]──▶ PC    PC 检测到 gap (expected=3, got=4)
         ◀──[SYNC_REQUEST]──  PC     PC 请求全量同步
  Phone ──[seq=5, FullSync]──▶ PC     手机发送输入框全文

设计原则:
  ✅ 不做逐包 ACK（避免延迟）
  ✅ 不做重传（旧 delta 已过时）
  ✅ 发现不一致 → FullSync（简单可靠）
```

### 心跳

| 心跳间隔 | 5 秒 |

### TEXT_DELTA (0x01) — 核心消息

**JSON 格式**：

```json
{
  "t": 1,           // type: TEXT_DELTA
  "s": 42,          // seq: 序列号
  "o": "A",         // op: A=Append, I=Insert, D=Delete, R=Replace
  "p": 0,           // position: 操作位置（字符偏移，APPEND 时忽略）
  "n": 0,           // delete_count: 删除字符数（仅 D/R 使用）
  "d": "你好世界",   // data: 文本数据
  "c": false         // clipboard_hint: 建议 PC 端使用剪贴板注入
}
```

**op 操作类型**：

| op | 含义 | position | delete_count | data | 使用场景 |
|----|------|----------|-------------|------|---------|
| `A` | 追加 | 忽略 | 忽略 | 追加的文本 | 用户在末尾输入（最常见，约 90%） |
| `I` | 插入 | 插入位置 | 忽略 | 插入的文本 | 用户把光标移到中间后输入 |
| `D` | 删除 | 删除起始位置 | 删除字符数 | 空 | 用户按退格键 |
| `R` | 替换 | 替换起始位置 | 被替换字符数 | 新文本 | 候选词替换、自动纠错 |

**clipboard_hint 触发规则**：

```
data.length > 10  →  c = true  （建议剪贴板注入）
data.length ≤ 10  →  c = false （逐字符 SendInput）
```

**场景映射示例**：

```
场景 A：拼音逐字输入 "你好世界"
  → {"t":1,"s":1,"o":"A","d":"你","c":false}
  → {"t":1,"s":2,"o":"A","d":"好","c":false}
  → {"t":1,"s":3,"o":"A","d":"世","c":false}
  → {"t":1,"s":4,"o":"A","d":"界","c":false}

场景 B：语音整句 "今天天气真不错我们去公园散步吧"
  → {"t":1,"s":5,"o":"A","d":"今天天气真不错我们去公园散步吧","c":true}

场景 C：候选词替换 "北京" → "南京"
  → {"t":1,"s":6,"o":"R","p":2,"n":1,"d":"南","c":false}

场景 D：尾部删除一个字
  → {"t":1,"s":7,"o":"D","p":5,"n":1,"d":"","c":false}

场景 E：变化超过 60%，触发全量同步
  → {"t":2,"s":8,"d":"完全不同的新内容"}
```

```json
{
  "t": 2,

# 文档 6：Low Level Design

## 6.1 手机端 Low Level Design

### 6.1.1 DiffEngine 类

**职责**: 计算两个文本状态之间的最小增量操作。

**算法**: 最长公共前缀 + 最长公共后缀，时间复杂度 O(N)。

```dart
/// 文件: lib/core/diff_engine.dart

enum DeltaOp { APPEND, INSERT, DELETE, REPLACE, FULL_SYNC, NO_CHANGE }

class TextDelta {
  final DeltaOp op;
  final int position;
  final int deleteCount;
  final String text;
  final bool clipboardHint;

  TextDelta({
    required this.op,
    this.position = 0,
    this.deleteCount = 0,
    this.text = '',
    this.clipboardHint = false,
  });
}

class DiffEngine {
  String _previousText = '';

  TextDelta computeDelta(String newText) {
    final old = _previousText;
    _previousText = newText;

    // 边界：空 → 有内容
    if (old.isEmpty && newText.isNotEmpty) {
      return TextDelta(
        op: DeltaOp.APPEND,
        text: newText,
        clipboardHint: newText.length > 10,
      );
    }

    // 边界：有内容 → 空
    if (newText.isEmpty && old.isNotEmpty) {
      return TextDelta(
        op: DeltaOp.DELETE,
        position: 0,
        deleteCount: old.length,
      );
    }

    // 核心 Diff：前缀 + 后缀
    int prefixLen = 0;
    final minLen = old.length < newText.length ? old.length : newText.length;

    while (prefixLen < minLen && old[prefixLen] == newText[prefixLen]) {
      prefixLen++;
    }

    // 完全相同
    if (prefixLen == old.length && prefixLen == newText.length) {
      return TextDelta(op: DeltaOp.NO_CHANGE);
    }

    // 最长公共后缀
    int suffixLen = 0;
    while (suffixLen < (minLen - prefixLen) &&
           old[old.length - 1 - suffixLen] ==
           newText[newText.length - 1 - suffixLen]) {
      suffixLen++;
    }

    // 提取变化部分
    final deletedLen = old.length - prefixLen - suffixLen;
    final insertedText = newText.substring(prefixLen, newText.length - suffixLen);

    // 变化超过 60% → 全量同步
    if (deletedLen > old.length * 0.6 && old.length > 5) {
      return TextDelta(
        op: DeltaOp.FULL_SYNC,
        text: newText,
        clipboardHint: newText.length > 10,
      );
